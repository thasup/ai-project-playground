The Fundamental Truth

Every LLM call is stateless.

Foundation models do not update their weights during a chat. They have zero built-in memory. To build a "stateful" conversation, you must explicitly inject the state payload into every API request.

Complexity Scale

Roadmap to Stateful AI

From naive implementations to production-grade state architectures.

L0
Full Context (Naive)
Send the entire state every time. Works until it hits the token wall.
L1
Sliding Window (FIFO)
Keep only the last N messages in state. Permanent amnesia for older turns.
L2
Summarization
Compress old turns into a summary digest to save context payload size.
L3
RAG (Retrieval-Augmented Generation)
Externalize state to Vector DB. Retrieve only what is semantically relevant.
INDUSTRY STANDARD
L4
Prompt Caching (Massive Context)
Send huge state payload but cache the computation state at the provider edge.
L5*
GraphRAG
Entity extraction mapped to relational Knowledge Graphs.
L6*
Agentic / Managed Memory
LLM autonomously manages its own state via integrated tools.
We focus on implementing Levels 0 through 4.
* Levels 5 & 6 represent autonomous workflows currently outside our explicit state management scope.

Setting Boundaries

Beyond Explicit State (Out of Scope)

Transparency Note
The AI industry is moving towards autonomous workflows. You may hear buzzwords about technologies beyond standard explicitly-managed context. These require specialized Data Engineering, Graph Databases, or Agentic systems. They are outside our standard Frontend explicitly-stateful scope (Level 5 & 6).
L5: GraphRAG
Instead of just semantic similarity, extracts entities into a Knowledge Graph (Neo4j) to allow multi-hop reasoning. Highly complex data pipelines.
L6: Agentic / Managed Memory
Systems like MemGPT, Mem0, or Zep where the LLM is given tools to autonomously read/write its own memory pages to a database (delegated state).
Critical Finding · Liu et al., Stanford 2023

State Positioning Matters

LLMs do not attend uniformly across their context window. Position determines recall probability.

"Lost in the Middle": Recall drops to ~32% at center. Model has recency bias (end) + primacy bias (start). Middle state is statistically likely to be ignored.
✗ WRONG APPROACH
−System → History → [RAG state in middle] → User
−Critical rules buried after long history
−Retrieval order = whatever DB returns first
✓ CORRECT STATE ORDERING
+Critical <rules> → START of context payload
+Most important RAG state → first or last
+Recent turns → END of context
+Low-confidence/older state → middle
◈ Context payload order is a performance variable. Treat it like z-index in CSS — intentional and systematic.

Summary

5 Things to Walk Away With

01
Memory = State Management
The foundation model doesn't remember anything. "Context Engineering" is explicitly injecting state payloads into a stateless API.
02
Prompts ⊂ Context
A prompt is just a named region inside the context window. Use explicit structural tags (like `<rules>` or `<skills>`) to structure your state.
03
State Positioning is a performance variable
'Lost in the Middle': recall drops to ~32% in the center. Critical state payloads belong at the start or end.
04
Match technique to constraints
Level 1 (Sliding Window) ships in hours. Level 3 (RAG) takes weeks. Don't over-engineer state architectures prematurely.
05
Inference-time compute is the 3rd lever
Extended thinking parameters let you trade execution time for reasoning quality after the state payload is sent.
